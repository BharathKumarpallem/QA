# Engineering Approach: AI Document Q&A Service

This document describes the design philosophies and functional approaches taken to implement the AI Document Q&A (RAG) Service.

---

### 1.1. Duplicate Detection
- **Mechanism**: Every document's full raw binary payload is hashed using **SHA-256**.
- **Lookup**: Before processing the PDF, ChromaDB is queried using a `where={"sha256": file_sha256}` filter.
- **Action**: If a matching record is found, ingestion is aborted, returning an HTTP `409 Conflict` containing the UUID of the pre-existing document. This saves computation and OpenAI embedding fees.

### 1.2. Scanned PDF Detection
- **Mechanism**: In a production environment, sending scanned PDFs (images without embedded text layers) to a standard text extractor returns empty string pages.
- **Rule**: We compute the average character length per page across the document. If it is less than `50` characters, we immediately raise a `ScannedDocumentException` (HTTP `422 Unprocessable Entity`) indicating that OCR is required.

### 1.3. Page-Aware Token-Based Chunking
- **Tokenizer**: We use the `cl100k_base` tokenizer (using `tiktoken`), matching the encoding structure of OpenAI's `text-embedding-3-small` model.
- **Boundary**: Chunks are constructed *within* each page's boundaries. This ensures that every chunk is explicitly mapped to a single, correct `page_number` in metadata, guaranteeing accurate citations.
- **Parameters**: Default chunk size of `500` tokens with a `50` token overlap. If the page contains fewer than 500 tokens, it is stored as a single chunk.

## 2. Retrieval & Semantic Search

### 2.1. Vector Representation
- Incoming queries are vectorized using OpenAI's `text-embedding-3-small` endpoint to retrieve a 1536-dimensional float vector.
- ChromaDB is configured to run search matches in the **cosine space** (`hnsw:space = cosine`).

### 2.2. Multi-Document Context Search
- Callers of both `/search` and `/qa` can pass an optional array of `document_ids`.
- If provided, ChromaDB uses metadata filtering `where={"document_id": {"$in": document_ids}}` (or a flat filter for single items) to restrict matching only to those files.

---

## 3. Question Answering & Hallucination Prevention

### 3.1. Confidence Threshold
- Before requesting answers from OpenAI, we evaluate the best similarity score (`1 - cosine_distance`) among the top retrieved chunks.
- If the best score is `< 0.55` (default setting), we abort the LLM request and return:
  > *"I cannot find the answer to your question in the uploaded documents."*
- This prevents the LLM from attempting to answer queries that have no relevant source context, eliminating hallucination risks.

### 3.2. Structured JSON Completeness
- We enforce structured responses from `gpt-4o-mini` using OpenAI's `beta.chat.completions.parse` method.
- The model is instructed to fit responses directly into our Pydantic JSON schema containing:
  - `answer` (str)
  - `confidence_score` (float)
  - `citations` (List of citation objects)

### 3.3. Citation Post-Validation
- Rather than blindly trusting the citations generated by the LLM, the backend runs a validation step:
  - We normalize casing and whitespaces of the LLM's `exact_snippet`.
  - We check if it is a verbatim substring of the actual source text retrieved from the database for the cited `document_id` and `page_number`.
  - Discrepancies are logged, and invalid citations are stripped. If all citations fail validation, the answer's confidence score is heavily penalized.
